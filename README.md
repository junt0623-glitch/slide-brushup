# スライドブラッシュアップ工房 ─ Phase 1・2・2.1・3

pptxインポート・解析・ハッシュ台帳・サムネイル表示（Phase 1）＋
出力エンジン・温存スライド注入・往復検証（Phase 2）＋
図形ツール認識・テーマ色解決・アニメーション/画面切り替え対応（Phase 2.1）＋
AI連携・全体構成分析・デザインシステム提案（Phase 3）。設計書§7に対応。

## Phase 3で追加されたこと

デッキ全体をAIに読み込ませ、①キーメッセージ ②分割/統合プラン ③反復スライド
挿入計画 ④デザインシステム案3つ、を一度に提案してもらう機能です。
まだ全ページへの適用（実際のブラッシュアップ）は行いません。それはPhase 5で、
ここで選んだ案に沿って行います。

**実行方法が2通りあります。**

| | 手動コピペモード（既定） | APIキー自動実行 |
|---|---|---|
| 費用 | **無料**（claude.aiの無料枠を利用） | 有料（1回あたり数十円程度） |
| 手間 | コピー&貼り付けが必要（画面の指示通りでOK） | ボタン1つで自動 |
| APIキー | 不要 | 必要 |

予算をかけたくない場合は、何もせず**手動コピペモードのまま**お使いください
（APIキーを設定していなければ自動的にこちらが選ばれます）。

### 使い方（手動コピペモード・無料）

1. 解析結果画面の「AIによる全体構成分析」セクションで「①のプロンプトをコピーする」を押す
2. [claude.ai](https://claude.ai) を新しいタブで開く（無料アカウントでOK。1日の
   メッセージ数に上限がありますが、この用途であれば十分な回数使えます）
3. 新規チャットにコピーした内容をそのまま貼り付けて送信
4. 返ってきた回答をコピーし、アプリの貼り付け欄に貼って「貼り付けた回答を読み込む」を押す
5. 続けて「②のプロンプトをコピーする」→同様にclaude.aiに貼り付け→回答を貼り付けて読み込む
6. キーメッセージ・プラン・デザイン案3つが表示されます

### 使い方（APIキー自動実行・有料）

1. 「⚙ 設定」でAnthropic APIキーを入力し、「APIキーで自動実行」を選択
   （APIキーは**このタブを閉じるまでの間だけ**ブラウザ内に保持されます。
   閉じたり再読み込みすると消えるため、都度入力が必要です。安全のための意図的な仕様です）
2. 「① ページ要約を生成する（Haiku）」→「② 全体構成を分析する（Fable 5）」の順に押す
   （②は1〜2分ほどかかります）

### どちらのモードでも

3つのデザイン案には、代表的な数ページに適用した場合の簡易プレビューが付きます。
気に入った案の「この案を採用する」を押すと、プロジェクトに保存されます
（後で選び直すことも可能です）。

### 正直な注意点

- **プロンプトの文言は暫定版です。** 実装引き継ぎ書§9に「文言の確定はFable 5で行う」と
  明記されている通り、今回は骨子（含めるべき情報・制約）に沿ってこちらで実装しましたが、
  言い回しの精度や結果の質に納得できない場合は、Fable 5セッションでプロンプト自体を
  見直すことをおすすめします（設定画面でモデルIDは自由に変更できるようにしてあります）
- **このリポジトリの自動テスト（bt09・bt10）は実際のAPIを呼びません。** APIキーをCI環境に
  持たせない方針のため、Claude APIへの通信をモックして「UIの配線」だけを検証しています。
  実際のAI応答の質は、手動コピペモードなどでご自身の目で確認してください
- デザインシステム案のプレビューは簡易版（見出し・本文・画像枠などの概略配置）です。
  実際の全ページ適用（Phase 5）ではもっと精密なspecが生成されます
- 全体構成分析（②）は、APIキー自動実行の場合のみ1回あたり利用料が発生します（Fable 5は
  高性能な分、単価も高めです。設計書§7の見積りでは入力25K/出力10Kトークン程度を想定）。
  手動コピペモードなら費用はかかりません

## Phase 2.1で修正・追加されたこと（ユーザー報告の不具合対応）

**① 図形ツールで作成した図形が消える不具合の修正**

原因: PowerPointの図形ツールで作った図形は、色を図形自身に持たず
`<p:style>` でテーマ色（accent1等）を参照します。以前の実装は図形自身の
明示的な色しか見ておらず、「色情報なしの図形」と誤判定して出力から
落としていました。

修正内容:
- テーマファイル（theme1.xml）のカラースキームを読み込み、テーマ色参照を
  実際の色（HEX）へ解決するようにした（lumMod/lumOff/tint/shade等の
  明度調整も近似計算で反映）
- `<p:style>` のfillRef/lnRef（テーマスタイル参照）を読むようにした
- テキストボックス/プレースホルダーと「本当の図形」を区別し、図形は
  色が解決できない場合でも必ず出力に残すようにした

**② アニメーション・画面切り替えへの対応**

正直な前提: レイアウトを組み替える「再構築」と、特定の図形に紐づいた
「アニメーション」の保持は、原理的に両立できません（アニメーションは
図形のIDを参照しており、再構築すると参照先が消えるため）。
そこで次の方針にしました。

- **アニメーションを含むスライドは、初期設定で「元のまま温存」**されます
  （アニメーションも完全に残ります）
- 各スライドのカードに**「元のまま温存する」チェックボックス**が付きました。
  チェックを外すとそのスライドは再構築の対象になります
  （アニメーションは失われ、書き出し時に警告が表示されます）
- グラフ・SmartArt等を含むスライドの温存は解除できません（再構築不能なため）
- **画面切り替え（スライド間のフェード等）は、再構築されたスライドにも
  自動で引き継がれます**（こちらは図形に紐づかないため保持可能です）

また、解析エンジンの更新に伴い、**旧バージョンで解析した保存済み
プロジェクトは書き出しボタンが無効化**され、再読み込みを促す警告が
表示されます（図形・アニメーション情報を持っていない古い解析データから
不完全なファイルを書き出してしまう事故を防ぐため）。

## Phase 2で追加されたこと

- **「pptxを書き出す」ボタン**: 解析結果（elements）からPptxGenJSで新しいpptxを再構築し、
  ダウンロードします。まだAIによるレイアウト再設計は行わず、**元と同じ内容・同じ配置**で
  1:1再構築することの正しさを先に固めるフェーズです（Phase 3以降でAIによる
  ブラッシュアップ提案が入ります）。
- **温存スライドの注入**: グラフ・SmartArt・埋め込みオブジェクト等を含むスライドは
  PptxGenJSで作り直さず、元のXMLパーツをZIPレベルでそのまま差し込みます
  （実装引き継ぎ書§5「最難関」の処理。グラフの系列データ等も完全に無傷で残ります）。
- **書き出し直後の自動検証**: 書き出したpptxを同じ解析エンジンで再度読み込み、
  ①スライド数一致 ②全文テキストハッシュ一致（無改変の証明） ③画像バイト一致
  ④温存スライドの再検出、の4点を実際にチェックして結果を表示します
  （「動くはず」ではなく実行結果で確認する方針）。
- **図形ツールの認識**（Phase 1からの修正）: テキストを持たない矩形・矢印などの
  装飾図形も、塗り色・線色・図形の種類（矩形／角丸／矢印など）込みで
  ブラッシュアップ対象として認識するようになりました。以前は「テキストを持つ図形」
  しか拾っていませんでした。

## 使い方（Phase 2）

1. Phase 1と同様にpptxを解析
2. 解析結果画面で「**pptxを書き出す（Phase 2）**」をクリック
3. 数秒〜数十秒後、検証結果が表示され、同時にファイルがダウンロードされます
4. 検証結果がすべて「OK」なら、ダウンロードされたpptxをPowerPointで開いて
   目視確認してください（グラフ・表・図形が壊れていないか等）

検証結果が1つでも「NG」の場合は、書き出し自体は完了していますが、無改変性が
崩れている可能性があります。「解析結果をJSONで書き出す」で元データと比較してください。

## Phase 2の既知の制約

- テキスト入りの図形（角丸四角形にラベル、等）は、PowerPoint上では
  「図形」と「テキストボックス」の2つの重なったオブジェクトとして再現されます
  （PptxGenJSの仕様上の制約。見た目は同じですが、後で動かす際は2つ選択が必要）
- ネストしたグループ内の回転は、グループ自身の回転との合成計算をしていません
  （単純な位置・サイズのグループ変換は正しく計算されます）
- EMF/WMF形式の画像は書き出し時にスキップされます（Phase 1同様、解析結果には
  バイトデータとして保持されていますが、pptxへの再埋め込みは非対応）
- コネクタ線（矢印線等）は直線として簡略化されます
- 再構築されたスライドは新しいデザイン（PptxGenJSの既定スタイル）になり、
  温存スライドは元の見た目のまま混在します。見た目の統一はPhase 3以降で
  AIによるデザイン提案が入った際に行われます（現時点では仕様どおりです）

## できないこと（Phase 1からの既知の制約・継続）

- **`.ppt`（97-2003形式）は非対応**。PowerPointで`.pptx`に変換してから使用
- レイアウト/マスターからのプレースホルダ位置継承は解決しない
- グラフ・SmartArtの中身（系列データや図形テキスト）は解析しない
  （温存対象として検出し、Phase 2で元データごと保持する設計のため）

## 動作要件

- **HTTPS環境で開くこと**（GitHub Pagesならそのまま満たされる）。
  `crypto.subtle`（ハッシュ計算）はセキュアコンテキストでのみ動作するため、
  `index.html` をローカルで直接ダブルクリックして `file://` で開くと動作しません。
- 対応ブラウザ: 最新のChrome / Edge / Safari（Windows / Mac / iPad）

## テストについて（GitHub Actions）

Phase 3で追加された回帰テスト:

- **bt09**: APIキー自動実行モードのUI配線を、Claude API呼び出しをモックして検証
  （APIキー未設定時のエラー表示／要約生成→全体構成分析→デザイン案選択→永続化の一連の流れ／
  401エラーやJSON解釈失敗時のエラーハンドリング）

  貼り付けた回答の読み込みが正しく反映されるか／**Anthropic APIへの通信が一切発生しない**こと）

bt01〜bt08と合わせて、GitHubへのpushまたはPull Requestで自動実行されます。
